## Setup

<Steps>

1. **Clone and install**

   ```bash
   git clone https://github.com/sasy-labs/sasy-demo.git
   cd sasy-demo
   make setup
   ```

2. **Configure your keys**

   ```bash
   cp .env.example .env
   ```

   The demo uses three API keys. Edit `.env` and fill them in:

   - `OPENAI_API_KEY` — powers the airline agent in the demo scenarios
     (the LLM acting as a customer-service rep). This is the demo's
     choice of agent LLM; it's unrelated to SASY policy enforcement.
   - `SASY_API_KEY` — authenticates to the SASY policy engine. Provided
     to you separately.
   - `ANTHROPIC_API_KEY` — **optional**. Only needed if you run
     `make translate` and want to use your own Anthropic credits instead
     of the shared server-side key (which is subject to a rate limit).

   ```bash
   OPENAI_API_KEY=sk-...
   SASY_API_KEY=demo-key-your-key-here
   # ANTHROPIC_API_KEY=sk-ant-...   (optional; see above)
   ```

</Steps>

## Translate a Policy

The demo above ran against a hand-written Datalog
policy (`policy.dl`) shipped with the repo — fine
for the walkthrough, but not how you'd author one
from scratch. For your own agent, the typical
workflow is to write the policy in **English** and
let SASY compile it to Datalog for you. Here's how:

Write your policy in English (plus point the
translator at your agentic codebase) and get back a
Soufflé Datalog policy, plus a companion C++
functors file if the policy needs custom helpers
(see below):

<Tabs>
  <TabItem label="SDK">
    ```python
    import logging
    from sasy.policy import translate

    # See progress while the job runs.
    logging.basicConfig(format="%(message)s")
    logging.getLogger("sasy").setLevel(logging.INFO)

    with open("policy_english.md") as f:
        policy = f.read()

    result = translate(
        policy,
        codebase_paths=["src/demo"],  # your agent
    )
    result.print_summary()
    result.save_all("output/", base_name="airline")
    ```
  </TabItem>
  <TabItem label="Makefile">
    ```bash
    make translate
    ```
  </TabItem>
</Tabs>

Translation takes ~5–15 minutes. Writes
`output/airline_policy.dl` and
`output/agent_summary.md`. A companion
`output/airline_functors.cpp` is written too if
the policy needs custom C++ helpers (content
matching, date arithmetic, NLP classifiers); the
demo policy is pure attribute matching, so no
functors file is emitted. The richer
[tau2 airline benchmark](/benchmarks/tau2-airline/)
policy does produce functors if you want to see
the feature in practice.

Upload and run the translated policy against the
scenarios:

```bash
make upload-translated    # ships output/airline_policy.dl
make demo-translated      # runs scenarios against it
```

See [Translate a Policy](/translate/) for the full
API and the [tau2 airline benchmark](/benchmarks/tau2-airline/)
for empirical results.

## Integrate SASY in Your Own Agent

Here's how to wire the SDK into your own agent code.

### 1. Configure

Set `SASY_API_KEY` (and optionally `SASY_URL`) in
`.env` — the SDK reads them automatically on first
use. To override at runtime (different endpoint
per environment, custom auth hook, etc.):

```python
from sasy.config import configure
from sasy.auth.hooks import APIKeyAuthHook

configure(
    url="sasy.fly.dev:443",
    auth_hook=APIKeyAuthHook(api_key="..."),
)
```

### 2. Upload

<Tabs>
  <TabItem label="SDK">
    ```python
    from sasy.policy import upload_policy_file

    resp = upload_policy_file("policy.dl")
    print("Uploaded!" if resp.accepted else resp.error_output)
    ```
  </TabItem>
  <TabItem label="Makefile">
    ```bash
    make upload              # uploads policy.dl (reference)
    make upload-translated   # uploads output/airline_policy.dl
    ```
  </TabItem>
</Tabs>

### 3. Check tool calls

Before executing any tool, check with the policy
engine. SASY tracks every recorded message + tool
call as a node in a **dependency graph**, and
policies can reason about prior context (e.g.
*"the agent must call `get_reservation_details`
before `cancel_reservation`"*). When you check a
tool call, pass the IDs of the prior events that
led up to it; the engine walks the graph to
evaluate guard rules.

```python
import json
from sasy.reference_monitor import check_tool_call

# args is a JSON string (matches what OpenAI / Anthropic
# tool-use APIs already give you in `tool_call.arguments`).
auth = check_tool_call(
    fn_name="cancel_reservation",
    args=json.dumps({"reservation_id": "RKLA42"}),
    # Pass the IDs of prior recorded events (user message,
    # tool results, etc.) that this call depends on. If you
    # haven't wired up observability yet, pass None — the
    # check still works, it just can't enforce graph guards.
    input_node_ids=None,
)

if auth.authorized:
    # ...invoke the tool the same way you would today...
    result = my_tools.cancel_reservation(reservation_id="RKLA42")
else:
    # Flat surface — see `auth.denial_trace` for the full
    # structured proto if you need reason codes / source
    # locations.
    print(f"Denied: {auth.denial_reasons}")
    print(f"Suggestions: {auth.suggestions}")
```

Every tool call your agent makes goes through
this check. The policy engine evaluates it against
the loaded Datalog and returns AUTHORIZED or
DENIED with a human-readable reason. See [How
Enforcement Works](/demo/enforcement/) for how
events are recorded into the graph.

<Aside>
  The demo code in `src/demo/agent.py`
  shows the full integration — how tool calls are
  recorded in the dependency graph and checked
  against the policy.
</Aside>

### 4. Instrument (optional)

For automatic instrumentation, one call patches
the supported frameworks (auth comes from `.env`):

```python
import sasy
sasy.instrument()
```

`sasy.instrument()` currently patches:

- `httpx` and `requests` (outbound HTTP — recorded
  + proxied through SASY for credential injection
  + policy checks)
- Langroid (`ChatAgent` tool calls — auto-checked
  before invocation)
- Tau2-bench agents (the user-simulator + assistant
  loops the airline benchmark uses)

After `instrument()`, your existing tool calls
flow through SASY without code changes:

```python
import sasy
import langroid as lr

sasy.instrument()

agent = lr.ChatAgent(lr.ChatAgentConfig(...))
agent.enable_message(MyTool)
# Tool calls inside agent.llm_response() are now
# checked against the loaded policy automatically;
# denied calls raise (or return a typed error,
# depending on the integration).
response = agent.llm_response("cancel my booking")
```

For a runnable end-to-end example see
`src/demo/agent.py` in this repo, which wires up
the airline scenarios with full graph recording.
